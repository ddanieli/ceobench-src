System Overview

SaaS Bench simulates running an AI SaaS company (NovaMind) over 3,650 days (10 years). An AI agent manages pricing, capacity, R&D, advertising, and negotiations to maximize founder dividends.

Main Simulation Loop

The agent interacts with the Gym-style environment: call tools to configure the business, then next_day to advance. Each day returns a dashboard with metrics, inbox, and equity info.

Customer Lifecycle

Customers flow through a lifecycle: lead generation (from ad channels + network effects) → participation curve evaluation → subscribe/lost → daily satisfaction updates → billing → churn/upgrade/downgrade.

sequenceDiagram
    participant Ads as 📢 Ad Channels
    participant Net as 🔗 Network Effects
    participant Gen as 🎯 Lead Generator
    participant Curve as 📊 Participation Curve
    participant DB as 💾 Database
    participant Sat as 😊 Satisfaction Engine
    participant Bill as 💰 Billing Engine

    Note over Ads,Bill: === Lead Generation ===

    rect rgba(88, 166, 255, 0.06)
    Ads->>Gen: social_media + search_ads + linkedin + content + referral
    Note right of Ads: Each channel: base_rate × spend^0.5 × group_reach × reputation
    Net->>Gen: Subscriber network referrals
    Note right of Net: NETWORK_INFLUENCE_MATRIX[source_group][target_group]
    Gen->>Gen: Apply weekly + monthly multipliers
    Gen->>Gen: Apply macro demand_multiplier per group
    Gen->>Gen: Poisson sample: leads ~ Poi(λ_total)
    end

    Note over Ads,Bill: === Per-Lead Evaluation ===

    rect rgba(63, 185, 80, 0.06)
    loop For each lead
        Gen->>Gen: _generate_customer_from_group(group_id)
        Note right of Gen: Sample: c_max, alpha, beta, steepness, q_min, q_max
        Gen->>DB: INSERT customer record

        Gen->>Curve: Evaluate all 3 plans (A, B, C)
        Note right of Curve: For each plan P:
Q = (base_quality + q_shared) × tier_mult
Satisfaction = sigmoid(Q, price, c_max, steepness)
Accept if Q ∈ [q_min, q_max] AND price ≤ budget

        alt Best plan found (satisfaction > 0)
            Curve-->>Gen: Best plan selected
            Gen->>DB: INSERT subscription (status='subscribed')
            Gen->>DB: UPDATE group_reputation += new_sub_bonus
        else No acceptable plan
            Curve-->>Gen: No plan acceptable
            Gen->>DB: UPDATE customer (status='lost')
        end
    end
    end

    Note over Ads,Bill: === Daily Satisfaction Updates ===

    rect rgba(210, 168, 255, 0.06)
    Sat->>DB: Batch-fetch all active subscribers
    loop For each subscriber
        Sat->>Sat: Compute Q_perceived
        Note right of Sat: Q = (base_pq + q_shared + q_group) × tier_mult
+ relationship_bonus + stickiness_bonus
− quota_penalty − overload_penalty − outage_penalty
        Sat->>DB: UPDATE customer_state.current_satisfaction
        Sat->>DB: UPDATE group_reputation (weighted by satisfaction)
    end
    end

    Note over Ads,Bill: === Monthly Billing (every 30 days) ===

    rect rgba(240, 136, 62, 0.06)
    Bill->>DB: Fetch subscribers at billing day
    loop For each billing subscriber
        Bill->>Curve: Re-evaluate best plan
        alt Still acceptable → stay/upgrade/downgrade
            Curve-->>Bill: Best plan (may differ from current)
            Bill->>DB: Process plan change if needed
            Bill->>DB: INSERT ledger (subscription_payment)
            Note right of Bill: payment = effective_price × seat_count
            Bill->>DB: INSERT ledger (overage_charge) if usage > quota
        else No plan acceptable → CHURN
            Curve-->>Bill: No acceptable plan
            Bill->>DB: UPDATE subscription (status='cancelled')
            Bill->>DB: UPDATE group_reputation -= churn_penalty
            Note right of Bill: Churn reasons: quality, price, budget, competitor, issue
        end
    end
    end

Enterprise Negotiation Flow

Enterprise customers (10-500 seats) arrive as leads and negotiate via a structured thread system. The agent sends offers; customers evaluate using their hidden preferences and counter-offer. Multi-turn negotiation with relationship tracking and deadlines.

sequenceDiagram
    participant Sim as ⚙️ Simulator
    participant Ent as 🏢 Enterprise Engine
    participant LLM as 🧠 Customer LLM
    participant Agent as 🤖 AI Agent
    participant DB as 💾 Database

    Note over Sim,DB: === Enterprise Lead Arrives ===
    Sim->>Ent: New enterprise customer (group: E1-E8)
    Ent->>DB: INSERT customer (type='large', seat_count=N)
    Ent->>Ent: create_negotiation_thread()
    Ent->>LLM: generate_enterprise_email(context: initial_inquiry)
    LLM-->>Ent: "We need AI for 50 seats, budget $X..."
    Ent->>DB: INSERT enterprise_turns (sender='customer', turn=0)

    Note over Sim,DB: === Inbox Notification ===
    Sim-->>Agent: Dashboard inbox: "📨 1 new enterprise lead (50 seats)"

    Note over Agent,DB: === Agent Responds with Offer ===
    Agent->>Agent: Analyze: group preferences, seat count, budget signals
    Agent->>Ent: send_enterprise_deal({thread_id, plan, price_per_seat, extras})
    Ent->>DB: INSERT enterprise_turns (sender='agent', turn=1)

    Note over Sim,DB: === Customer Evaluates (next step_day) ===
    rect rgba(210, 168, 255, 0.06)
    Sim->>Ent: _process_enterprise_negotiations(config)
    Ent->>Ent: get_threads_needing_reply()
    loop For each thread with agent reply
        Ent->>Ent: evaluate_offerings(agent_offer, customer_prefs)
        Note right of Ent: For each offering:
Q = get_quality_for_plan(plan, tier, q_shared)
satisfaction = sigmoid(Q, price×seats, c_max, steepness)
Compare to customer's best alternative

        alt Satisfaction ≥ threshold → ACCEPT
            Ent->>DB: UPDATE subscription (status='subscribed')
            Ent->>DB: UPDATE enterprise_turns (closed=1, outcome='accepted')
            Ent->>DB: UPDATE group_reputation (positive)
            Note right of Ent: 🎉 Deal closed! Revenue += price × seats
        else Satisfaction close but not enough → COUNTER-OFFER
            Ent->>Ent: compute_customer_counter_offer()
            Note right of Ent: Counter: lower price or suggest different plan
            Ent->>LLM: generate_enterprise_email(context: counter_offer)
            LLM-->>Ent: "We like the product but need a better price..."
            Ent->>DB: INSERT enterprise_turns (sender='customer')
            Ent->>DB: UPDATE relationship score
        else Satisfaction too low → REJECT
            Ent->>DB: UPDATE enterprise_turns (closed=1, outcome='rejected')
            Ent->>DB: UPDATE group_reputation (negative)
            Note right of Ent: 💔 Deal lost
        end
    end

    Note over Ent,DB: Deadline & Relationship Management
    Ent->>Ent: Check thread deadlines
    opt Thread expired (no agent reply in time)
        Ent->>DB: Close thread (outcome='expired')
        Ent->>DB: UPDATE group_reputation (deadline_penalty)
    end
    Ent->>Ent: Decay relationships on idle threads
    end

VC Investment Flow

Predefined VCs approach probabilistically. Each has unique valuation weights, check size ranges, and term sheet preferences. Multi-turn negotiation with anti-dilution, milestone tranching, and redemption rights. Settled deals create shareholders who receive dividends.

sequenceDiagram
    participant Sim as ⚙️ Simulator
    participant VC as 🏦 VC Engine
    participant Agent as 🤖 AI Agent
    participant DB as 💾 Database

    Note over Sim,DB: === VC Approach (daily probability roll) ===
    Sim->>VC: _generate_vc_approaches(config)
    loop For each predefined VC
        VC->>VC: Roll: random() < daily_approach_prob?
        opt VC approaches
            VC->>VC: _compute_vc_valuation(vc_profile)
            Note right of VC: Valuation = base_mult × ARR × weighted_score × macro
10 dimensions: growth, retention, margin, scale,
quality, market, efficiency, momentum, runway, diversity
            VC->>VC: Sample equity_pct in [min, max]
            VC->>VC: Assign term sheet items
            Note right of VC: Anti-dilution floor? Milestone tranching?
Redemption rights? Each probabilistic.
            VC->>DB: CREATE vc_approach record
            VC->>DB: INSERT vc_turns (sender='vc', initial offer)
        end
    end

    Note over Agent,DB: === Agent Sees VC in Inbox ===
    Sim-->>Agent: Dashboard: "🏦 1 new VC message"
    Agent->>Agent: Evaluate: check size, equity ask, term sheet, valuation

    alt Agent accepts/counter-offers
        Agent->>VC: send_vc_deal({shareholder_id, equity_pct, check_size, terms})
        VC->>DB: INSERT vc_turns (sender='agent')
    else Agent rejects
        Agent->>VC: reject_vc_deal({shareholder_id, reason})
        VC->>DB: Close negotiation (outcome='rejected_by_agent')
    end

    Note over Sim,DB: === VC Evaluates (next step_day) ===
    rect rgba(63, 185, 80, 0.06)
    Sim->>VC: _process_vc_negotiations(config)
    loop For each active VC thread with agent reply
        VC->>VC: evaluate_agent_vc_offer()
        Note right of VC: Compare agent's offer to VC's fair valuation
Check: equity_pct × valuation ≥ check_size?
Term sheet friendliness score

        alt Offer acceptable → SETTLE
            VC->>DB: CREATE shareholder record
            VC->>DB: Issue new shares (dilute founder)
            VC->>DB: INSERT ledger (vc_investment, +check_size)
            opt Has milestone tranching
                Note right of VC: Tranche 1 now, Tranche 2 on milestone
            end
            Note right of VC: 💰 Cash += check_size, Founder % decreases
        else Close but not enough → COUNTER
            VC->>VC: Compute counter-offer
            VC->>DB: INSERT vc_turns (sender='vc', counter)
        else Too far apart → WALK AWAY
            VC->>DB: Close negotiation (outcome='vc_walked')
        end
    end
    end

    Note over Sim,DB: === Term Sheet Triggers (ongoing) ===
    rect rgba(255, 123, 114, 0.06)
    Sim->>VC: _process_vc_term_triggers(config)
    opt Anti-dilution triggered
        Note right of VC: If company valuation drops below floor
→ Investor gets extra shares at no cost
        VC->>DB: UPDATE shareholder shares (increase)
    end
    opt Milestone reached
        Note right of VC: Revenue milestone hit
→ Release Tranche 2
        VC->>DB: INSERT ledger (vc_investment, +tranche_2)
    end
    opt Redemption triggered
        Note right of VC: Deadline passed, VC exercises redemption
→ Company must buy back at multiplier
        VC->>DB: INSERT ledger (vc_redemption, −buyback_amount)
    end
    end

    Note over Sim,DB: === Dividends (Agent-initiated) ===
    rect rgba(88, 166, 255, 0.06)
    Agent->>DB: declare_dividend(amount)
    DB->>DB: Distribute pro-rata to all shareholders
    Note right of DB: Founder + all VCs get shares_held/total_shares × amount
    DB->>DB: INSERT dividends records
    Note right of DB: 📈 Score = SUM(founder_payout) FROM dividends
    end

Macroeconomic Cycle

The PMI (Purchasing Manager's Index) follows an Ornstein-Uhlenbeck process with sinusoidal mean-reversion. It affects customer demand, willingness to pay, churn tolerance, and seat counts differently per group via MACRO_SENSITIVITY config.

sequenceDiagram
    participant Sim as ⚙️ Simulator
    participant PMI as 📈 PMI Engine
    participant Groups as 👥 Customer Groups
    participant DB as 💾 Database

    Note over Sim,DB: === PMI Update (every N days) ===

    rect rgba(88, 166, 255, 0.06)
    Sim->>PMI: _process_macroeconomic_cycle(config)
    PMI->>PMI: Ornstein-Uhlenbeck step
    Note right of PMI: dPMI = θ(μ(t) − PMI)dt + σ·dW
μ(t) = 50 + A·sin(2π·t/P + φ)
θ = mean-reversion speed
σ = volatility
    PMI->>PMI: Clamp PMI to [25, 75]
    PMI->>PMI: Buffer for delayed publication
    Note right of PMI: PMI published after measurement_lag days
    end

    rect rgba(63, 185, 80, 0.06)
    Note over PMI,Groups: === Compute Per-Group Multipliers ===
    PMI->>PMI: pmi_deviation = (PMI − 50) / 50
    loop For each customer group
        PMI->>Groups: Read MACRO_SENSITIVITY[group_id]
        Note right of Groups: Each group has 5 sensitivity parameters:
• demand (lead volume)
• willingness_to_pay (budget)
• churn_tolerance (patience)
• seat_count (enterprise sizing)
• price_sensitivity (price elasticity)

        Groups->>Groups: multiplier = 1.0 + sensitivity × pmi_deviation
